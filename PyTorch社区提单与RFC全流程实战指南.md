## 第一阶段：RFC 撰写

### 1.1 你的想法是否需要 RFC？

在动笔之前，先用三个问题自检：

| 问题 | 回答"是"→ 需要 RFC |
|---|---|
| 是否引入新的抽象层或核心概念？ | ✅ 走 RFC |
| 是否涉及 API 破坏性变更（breaking change）？ | ✅ 走 RFC |
| 是否跨多个模块（如同时影响 autograd + nn + distributed）？ | ✅ 走 RFC |
| 是否只是一个 bug 修复或文档改进？ | ❌ 直接提 PR |
| 是否是一个小的 API 增强（如加一个参数）？ | ❌ 提 Issue + dev-discuss 讨论即可 |

### 1.2 RFC 撰写指南

PyTorch 官方提供了 RFC 模板（`RFC-0000-template.md`），包含 9 个章节，可作为结构参考。但在实际撰写时，**不必完全遵从模板**，核心是把方案的逻辑讲清楚，让维护者和社区能快速理解你的意图。

> **📌 核心原则：从社区受益的角度出发，进而达成自己的目标。**
>
> RFC 的本质是说服社区"这个改动对 PyTorch 生态有价值"。你的方案可能服务于自家业务场景，但提案的论证逻辑应该是：**这个痛点不是只有你有，而是广泛存在于社区中** → **你的方案能系统性地解决这类问题** → **社区采纳后将惠及更多用户和场景**。站在社区的立场上论证方案，才能真正推动提案通过。
>
> 参考一个成功的 RFC 案例：[torch.unravel_index](https://github.com/pytorch/pytorch/issues/185590)——从 NumPy 对标、社区用户高频需求出发，简洁清晰地论证了 API 设计的合理性。

以下为模板各章节的意图说明，按需选用：

```
RFC 文件结构（按撰写顺序）
═══════════════════════════════════════════════════════════════

1. SUMMARY（摘要）
   ✎ 用 3-5 句话/要点说明你要做什么
   ⚠️ 关键：让维护者在 30 秒内理解你的提案是什么
   示例："本 RFC 提议为 torch.nn 添加一个 WeightOnlyQuantizedLinear
         层，用于支持 INT4/INT8 权重量化推理，内存占用降低 75%。"

2. MOTIVATION（动机）
   ✎ 为什么这个提案很重要？不做的代价是什么？
   ⚠️ 关键：用数据和场景说话，不要只说"我觉得有用"
   要点：
   - 解决了用户的什么痛点？
   - 有多少用户/场景会受益？
   - 如果不做，用户现在的 workaround 是什么？有多痛苦？
   - 与 PyTorch 的战略方向是否一致？

3. PROPOSED IMPLEMENTATION（方案设计）
   ✎ RFC 的核心部分，需要足够详细
   ⚠️ 关键：要让"熟悉 PyTorch 的人能理解设计，熟悉实现的人能直接编码"
   必须包含：
   - API 设计（含完整代码示例，展示用户如何使用）
   - 内部架构图/数据流（文字描述或 ASCII 图都可以）
   - 边界情况处理（空输入、极端值、不同 dtype/device 组合）
   - 与现有功能的交互（会和 torch.compile、autograd、Distributed 怎么交互？）
   - 新术语定义（如果你引入了新概念）

4. METRICS（衡量指标）
   ✎ 如何衡量这个 feature 的价值？
   示例：
   - 性能：forward 延迟降低 X%，内存占用降低 Y%
   - 使用率：预期被 Z 个下游项目采用
   - 准确性：在 benchmark A 上精度不低于 baseline

5. DRAWBACKS（风险与代价）
   ✎ 诚实地评估"为什么我们不该做这个"
   ⚠️ 这是维护者非常看重的一项，体现你的技术成熟度
   至少覆盖：
   - 是否是 breaking change？（如果是，迁移方案是什么？）
   - 代码复杂度增加多少？维护负担多大？
   - 与现有/规划中功能的冲突
   - 对用户体验的影响（API 是否会变得更复杂？）

6. ALTERNATIVES（替代方案）
   ✎ 你考虑过哪些其他设计？为什么不选它们？
   ⚠️ 展示你做了充分调研，不是一拍脑袋的方案
   包含：
   - 方案 A：xxx（优点/缺点）
   - 方案 B：xxx（优点/缺点）
   - "什么都不做"的后果是什么？

7. PRIOR ART（先例调研）
   ✎ 其他框架/库有没有类似功能？经验教训是什么？
   参考来源：
   - TensorFlow / JAX / MXNet 的对应功能
   - 学术论文中的实现
   - 其他语言/生态的做法
   ⚠️ 既要说好的经验，也要说失败的教训

8. HOW WE TEACH THIS（教学计划）
   ✎ 用户怎么学习这个新功能？
   要点：
   - 命名和术语是否直观？
   - 是否需要新增文档章节或重组现有文档？
   - 如何向现有 PyTorch 用户教学？（示例代码、教程、博客？）

9. UNRESOLVED QUESTIONS（待解决问题）
   ✎ 诚实地列出你还不太确定、需要在 RFC 讨论中解决的问题
   分类：
   - 需要在 RFC 合并前解决的
   - 可以在实现阶段解决的
   - 明确排除在本次 RFC 范围之外的
```

## 第二阶段：RFC 评审流程

RFC 评审分为两步：**先内部评审，通过后再提交社区评审**。这样可以在内部消化掉大部分设计问题，减少在社区中的迭代轮次和耗时。

### 2.1 第一步：内部评审

在 RFC 草案完成后，先在自己的团队或组织内部进行评审：

```
内部评审流程
═══════════════════════════════════════════════════════════════

Step 1: 内部宣讲方案
  ↓   向团队内有 PyTorch 经验的同事讲解 RFC 方案
  ↓   重点验证：方案逻辑是否自洽？是否有明显的遗漏或漏洞？

Step 2: 对照社区标准自查
  ↓   站在维护者的视角审视：
  ↓   - 方案的受益面是否足够广（不只是你的场景）？
  ↓   - API 设计是否简洁、符合 PyTorch 惯用风格？
  ↓   - 有没有充分考虑 Drawbacks 和 Alternatives？

Step 3: 修订至内部一致通过
  ↓   内部评审达成共识后，才进入社区评审阶段
  ↓   ⚠️ 如果内部都还有分歧，提交到社区只会放大问题
```

### 2.2 第二步：社区评审

内部评审通过后，正式提交到 PyTorch 社区进行公开评审。

#### 2.2.1 精准定位审阅者（最重要的一步）

不是你随便 @ 一个人就有人理你。必须精准定位：

```
Step 1: 查 CODEOWNERS 文件
  ↓   https://github.com/pytorch/pytorch/blob/master/CODEOWNERS
  ↓   找到你改动涉及的文件路径 → 对应的 owner

Step 2: 查 Persons of Interest 页面
  ↓   https://docs.pytorch.org/docs/2.12/community/persons_of_interest.html
  ↓   找到你涉及的模块 → 对应的维护者及其 GitHub ID

Step 3: 在 RFC PR 描述中 @ 对应维护者
  ↓   格式：cc @github_username

Step 4: 多重通知（广播你的 RFC）
  ↓   ① dev-discuss 论坛 RFC Chatter 分类下发帖
  ↓   ② Slack 频道（如有权限）提及
  ↓   ③ 在 pytorch/pytorch 主仓库开 issue 链接 RFC PR
```

### 2.2.2 联系维护者的沟通模板

**在 RFC PR 描述中：**

```markdown
## RFC: [标题]

RFC PR: pytorch/rfcs#XX

### 涉及模块
- torch.nn → cc @gchanan @jbschlosser
- torch.autograd → cc @ezyang @albanD

### 摘要
[一句话说明]

### 请求审阅的重点
特别希望就以下方面获得反馈：
1. API 设计是否合理？
2. 与现有 autograd 机制的交互是否有遗漏？
3. 性能方面的潜在风险？

### 相关讨论
- dev-discuss 帖子: [链接]
- 主仓库 issue: pytorch/pytorch#XXXX
```

### 2.3 如果没有人回应怎么办？（升级路径）

PyTorch 有明确的升级路径，不要默默地等：

```
Day 0:     提交 RFC / PR
Day 2-3:   在 PR 下礼貌地 ping 审阅者（@username 请帮忙看看）
Day 4:     ↓ 仍无回应
           双管齐下：
           ① 在 GitHub 上联系给你 PR 打 triage 标签的人
           ② 在 Slack 中直接联系对应 maintainer
              （前提：已在 Slack 中提供 GitHub ID 和 Gmail 邮箱且被邀请加入工作区）
Day 5+:    ↓ 仍无回应
           参加 Dev Infra Office Hours（每周五举行）
           当面提问，这是最高效的解决方式
           https://github.com/pytorch/pytorch/wiki/Dev-Infra-Office-Hours
           ↓ 仍无回应
           向 DevX 团队提 issue，描述你的阻塞情况
```

> **关于 Slack 联系 maintainer**：相比于 GitHub 上的异步审阅，Slack 是更高时效性的渠道。但需要提前完成两个前置条件：① 在 PyTorch Slack 工作区中提供你的 **GitHub ID** 和 **Gmail 邮箱**；② 被对应工作区的管理员邀请加入。满足条件后，可以在相关频道中直接 @ 对应的 maintainer 沟通，往往比等待 GitHub 审阅更快得到响应。

## 第三阶段：何时开始写代码？

这是社区开发者最容易踩坑的地方，用这张决策表：

```
┌────────────────────────────────────────────────────────────────┐
│                   何时开始写代码？决策树                         │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  你的提案是 bug fix 吗？                                        │
│    ├── 是 → 直接写代码提 PR ✅                                  │
│    └── 否 ↓                                                    │
│                                                                │
│  你的提案是小的、无争议的改进吗？                                  │
│  （如：加一个参数、修复类型标注、性能微调）                         │
│    ├── 是 → Issue 上快速讨论 → 写代码提 PR ✅                    │
│    └── 否 ↓                                                    │
│                                                                │
│  你的提案涉及新 API / 新抽象 / 架构变更吗？                       │
│    ├── 是 → ⚠️ 先 RFC，等 Accepted 后再写代码                    │
│    │       （否则写了也可能被推翻重来，浪费大量时间）               │
│    └── 否 ↓                                                    │
│                                                                │
│  不确定？                                                       │
│    └── → 在 dev-discuss 发帖讨论，问维护者是否需要 RFC            │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ ⚠️ 血泪教训：不要在 RFC 被接受之前就开始大规模编码！      │  │
│  │ 维护者可能要求完全不同的设计方案，你的代码可能全部作废。   │  │
│  │ 可以先写 prototype 验证可行性，但不要投入生产级代码。     │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 第四阶段：代码检视（Code Review）全流程

### 4.1 PyTorch PR 完整生命周期

```
提交 PR
  │
  ▼
Triage（~2 个工作日）
  → AutoLabel Bot 自动打标签
  → Triage 团队验证有效性 + 指派审阅者
  │
  ▼
CI 运行 + Review
  → CI 测试并行运行
  → 审阅者进行 Code Review
  │
  ├──→ 💬 Comment：非阻塞性意见，由作者自行决定是否采纳
  ├──→ ✅ Approve：批准（仅需 1 个 Approve 即可进入合并流程）
  └──→ ❌ Request Changes：阻塞性修改要求，必须先解决才能合并
        │
        ▼
      作者修改代码 → 推送更新 → 审阅者应在 24 小时内重新审阅
        │
        ▼
批准 + CI 通过
  │
  ▼
@PyTorchBot merge（作者或审阅者发起）
  │
  ▼
Trunk Tests（~6-8 小时，更全面的测试）
  │
  ▼
合并 ✅
```

> **⚠️ 关于 CI 门禁触发权限**：PyTorch 的 CI 流水线并非所有贡献者都可以触发，通常只有具有特定权限的成员（如 maintainer 或 triage 团队成员）才能启动完整的 CI 门禁。因此，在本地充分验证 + 存量检视意见修复完毕后，可以**找有权限的人（如给你审阅的 maintainer）帮忙触发 CI**，不要自己干等。

### 4.2 PyTorch Code Review 审阅者关注什么？

根据 [Code Review Values](https://github.com/pytorch/pytorch/wiki/Code-review-values)，审阅者会从以下维度检查：

| 维度 | 检查要点 |
|---|---|
| **PR 描述** | 是否说清楚了"为什么"做这个改动？描述和实际改动是否一致？ |
| **工程质量** | 是否保持向后兼容？是否用了惯用写法而非重复造轮子？有没有隐藏的假设可能导致未来出 bug？这个 feature 值得它的维护成本吗？ |
| **文档** | 新 feature 有文档吗？非显而易见的逻辑有注释吗？ |
| **测试** | 同一 diff 里有测试吗？测试覆盖了边界情况吗？（contiguous/non-contiguous、不同 dtype、空输入等）对难以测试的改动，说明了如何手动验证吗？ |
| **数值计算** | kernel 是 bit-for-bit 确定的吗？一次性操作是否移到了关键循环外？能否处理 >4GB 的数据？kernel launch 后是否调用了 `cudaGetLastError()`？ |

---

## 第五阶段：如何回复检视意见

### 5.1 三种审阅意见的应对策略

```
┌─────────────────────────────────────────────────────────────────┐
│                    审阅意见类型 & 应对策略                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│ 💬 Comment（非阻塞性建议）                                       │
│   审阅者的态度： "建议这样做，但你自己决定"                        │
│   你的策略：                                                     │
│   - 如果同意 → 修改，回复 "Done, thanks!"                        │
│   - 如果不同意 → 礼貌解释原因，但不一定要改                       │
│   - 如果不确定 → 回复 "Let me think about this" 然后认真评估      │
│                                                                 │
│ ❌ Request Changes（阻塞性修改要求）                              │
│   审阅者的态度： "这个必须改，否则不能合并"                        │
│   你的策略：                                                     │
│   - 理解审阅者的关切（即使不同意，也要先理解）                     │
│   - 如果同意 → 立即修改 + 回复修改内容概要                        │
│   - 如果不同意 → 用代码/数据/场景论证你的立场                     │
│   - 如果僵持 → 请第三方审阅者介入（升级路径见 2.3）                │
│                                                                 │
│ ✅ Approve（批准）                                                │
│   审阅者的态度： "可以合并了"                                     │
│   你的策略：                                                     │
│   - ⚠️ 先读完所有 comment！审阅者可能在 Approve 时附带小建议      │
│   - 处理完剩余小建议后再 @pytorchbot merge                        │
│   - 不要一看到 approve 就急着合并                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 回复审阅意见的沟通模板

```markdown
### 对于同意的修改建议：
> "Done. Added boundary check for empty tensor case in the latest commit (abc1234)."

### 对于不同意的建议：
> "Thanks for the suggestion! I considered this approach, but I think the
> current implementation is preferable because:
> 1. [具体理由，最好有代码/数据支撑]
> 2. [引用项目中已有的类似写法作为先例]
>
> What do you think? Happy to discuss further."

### 对于不理解/需要澄清的建议：
> "I'm not sure I fully understand the concern here. Could you elaborate on
> what scenario this might break? I'd like to make sure I address the root
> issue, not just the symptom."

### 当审阅者之间意见冲突时：
> "It seems like there are two different perspectives here:
> - @reviewer1 suggests approach A (reason: ...)
> - @reviewer2 suggests approach B (reason: ...)
>
> Could we get a third opinion to help resolve this?"
```

### 5.3 回应检视意见的黄金法则

```
┌────────────────────────────────────────────────────────────┐
│  📌 回应检视意见的四大黄金法则                               │
├────────────────────────────────────────────────────────────┤
│ 1. 不要有防御心理 —— 审阅者是在批评代码，不是在批评你        │
│ 2. 每个 comment 都要回复 —— 哪怕只是一个 👍 或 "Done"        │
│    这表示你认真看过了每一条意见                               │
│ 3. 修改后推送新 commit，不要 force push 覆盖旧 commit        │
│    （让审阅者能看到 incremental diff，便于重新审阅）          │
│    也不要 squash/合并 commit —— 每次修改独立保留，            │
│    检视人员可以直接看最新 commit 的改动点                     │
│ 4. 审阅者请求修改后应在 24h 内重新审阅 ——                    │
│    如果超过 2-3 天没回音，礼貌地 ping 一下                   │
└────────────────────────────────────────────────────────────┘
```
> **⏱️ 回复时效**：检视意见应**尽快回复**，不要让审阅者长时间等待。快速响应不仅能加速合并流程，也体现了你对贡献的投入程度。如果某个问题需要较长时间研究，可以先回复 "Looking into this, will update soon" 让审阅者知道你没有消失。

> **🤖 关于 AI 检视意见**：PyTorch 社区的 maintainer 可能会使用 AI 工具对 PR 进行辅助检视，这些 AI 生成的意见同样会出现在 PR 的 comment 中。**AI 意见也需要逐条认真回复**——它的本质和人类审阅者的 comment 一样，代表了对代码质量的关切。忽略 AI 意见等同于忽略审阅意见，会阻塞你的 PR。

## 第六阶段：高效沟通策略

### 6.1 沟通渠道选择矩阵

```
┌─────────────────────────────────────────────────────────────────┐
│                     沟通渠道选择指南                              │
├──────────────┬──────────────────┬────────────────────────────────┤
│ 渠道          │ 适用场景          │ 响应时效                        │
├──────────────┼──────────────────┼────────────────────────────────┤
│ GitHub PR    │ 代码级讨论、       │ 2-4 个工作日                    │
│ Comments     │ 技术细节、决策记录  │                                │
├──────────────┼──────────────────┼────────────────────────────────┤
│ dev-discuss  │ 设计方案讨论、     │ 1-3 天                          │
│ 论坛         │ RFC 传播、意见征集  │                                │
├──────────────┼──────────────────┼────────────────────────────────┤
│ Office Hours │ 紧急解阻塞、       │ 实时（每周五）                   │
│              │ 复杂问题当面讨论    │                                │
├──────────────┼──────────────────┼────────────────────────────────┤
│ Slack        │ 快速问答、         │ 分钟-小时级                     │
│ （需邀请）    │ 非正式讨论         │                                │
├──────────────┼──────────────────┼────────────────────────────────┤
│ GitHub Issue │ 跨 PR 讨论、      │ 1-3 天                          │
│ (主仓库)     │ 范围蔓延的议题      │                                │
└──────────────┴──────────────────┴────────────────────────────────┘
```

### 6.2 让维护者愿意回复你的技巧

```
✅ DO（应该做的）                          ❌ DON'T（不该做的）
─────────────────────────────────    ─────────────────────────────────
• PR 尽量小（<200 行），拆成多个        • 提交一个 3000 行的巨型 PR
  小 PR 依次提交                         （审阅者看到会关掉）

• 在 PR 描述中写清楚"为什么"            • 只写"fix bug"或"add feature"
  以及你做的设计决策和权衡               （维护者不知道从何看起）

• 自己做一次 self-review，标注出         • 把明显的格式问题、未完成代码
  你觉得有争议或需要重点审阅的部分        丢给审阅者（浪费双方时间）

• 提交前跑通本地 CI（lint + test）       • CI 全红就提交
                                         （维护者不会审阅失败的 PR）

• 一个 PR 只做一件事                     • 在同一个 PR 里混入重构+
                                         新功能+bug修复

• 用 Draft PR 表示"还在施工中，          • Draft PR 下频繁 @ 审阅者
  不要审阅"                              （Draft = 别看我）

• 超过 4 个工作日无回应才 ping           • 每天 ping 一次
                                         （维护者是志愿者，会反感）

• 周五下午不提 PR                        • 在维护者休假/周末时催促
  （大概率要到下周一才有人看）
```

### 6.3 异步沟通的核心原则

由于 PyTorch 维护者分布在全球不同时区（Meta 在美国西岸、NVIDIA 遍布全球、AMD 在美国、华为在中国等），异步沟通是常态：

```
┌────────────────────────────────────────────────────────────┐
│  📌 异步沟通效率最大化                                       │
├────────────────────────────────────────────────────────────┤
│ • 每条消息自包含 —— 让审阅者不需要回溯 50 条聊天记录          │
│   就能理解你的问题                                          │
│                                                            │
│ • 预判问题提前回答 —— 在描述中主动解释"为什么选 A 不选 B"     │
│   可以减少一轮异步往返                                      │
│                                                            │
│ • 利用时差 —— 你在东八区，维护者在西八区                      │
│   晚上提交代码 → 第二天早上看审阅意见 → 白天修改 → 循环       │
│                                                            │
│ • 一个往返的沟通最理想 —— 争取每次更新都让审阅者               │
│   只需看一遍就能做决定                                      │
└────────────────────────────────────────────────────────────┘
```
